# FS → TDS 技能

## 概述

使用AI辅助将功能规格说明书(FS)转换为技术设计规格说明书(TDS)。此技能自动化了从业务需求到技术实现细节的转换过程。

## 使用场景

- 使用功能规格文档开始新的开发周期
- 需要从业务需求创建技术设计
- 期望在整个项目中保持TDS格式标准化

## 输入要求

### FS文档结构（预期格式）
```markdown
# 功能规格说明书

## 1. 项目概述
## 2. 功能需求
### 2.1 功能点1
### 2.2 功能点2
## 3. 非功能需求
## 4. 技术架构
## 5. 验收标准
```

### 必需信息
- 项目名称
- 功能列表
- 技术栈偏好
- 非功能需求

## 阶段1：FS分析

### 步骤1.1：提取功能需求

使用AI解析并提取：
- 功能列表及优先级(P0/P1/P2)
- 用户故事
- 验收标准
- 业务规则

```bash
# 示例：使用AI提取功能
cat FS.md | grep -E "^### |^## " | head -20
```

### 步骤1.2：识别技术组件

针对每个功能需求，识别：
- 所需的API端点
- 所需的数据模型
- 外部依赖
- 集成点

### 步骤1.3：确定技术栈

根据FS需求确定：
- 前端：React、Vue、Next.js？
- 后端：FastAPI、Express、Spring？
- 数据库：PostgreSQL、MongoDB、SQLite？
- AI框架：LangChain、LlamaIndex？

## 阶段2：TDS生成

### 步骤2.1：创建TDS模板

```markdown
# 技术设计规格说明书 (TDS)

## 1. 项目概述
- 项目名称: [来源于FS]
- 功能简述: [来源于FS]
- 关联FS: [FS引用]

## 2. 技术方案
- 技术栈选型: [识别的技术栈]
- 架构设计: [架构图]
- 数据模型: [数据模型]

## 3. 接口设计
- API接口列表
- 请求/响应格式定义

## 4. 实现步骤
- 步骤N: [具体步骤]
- 验收条件: [每个步骤的验收条件]

## 5. 验收标准
- [ ] 功能点1
- [ ] 功能点2

## 6. 边界与约束
- 性能要求
- 安全要求
```

### 步骤2.2：生成实现步骤

针对FS中的每个功能：
1. 识别技术依赖
2. 拆分为可操作的步骤
3. 定义验收标准
4. 评估复杂度

### 步骤2.3：定义数据模型

生成：
- 数据库Schema
- API请求/响应模型
- 实体关系图

## 阶段3：验证

### 步骤3.1：完整性检查

验证FS的所有功能都在TDS中有对应：
- [ ] 所有P0功能有实现步骤
- [ ] 所有P1功能有实现步骤
- [ ] 已处理非功能需求
- [ ] 已识别技术约束

### 步骤3.2：技术评审

验证：
- 技术栈可行性
- 集成复杂度
- 性能影响
- 安全考虑

### 步骤3.3：需求分析师审批

向需求分析师展示TDS以获得签署：
- 功能覆盖率已验证
- 技术方案已批准
- 时间估算已验证

## 输出产物

| 产物 | 位置 | 格式 |
|------|------|------|
| TDS文档 | `requirements/TDS-{id}.md` | Markdown |
| API规范 | `docs/api/spec.yaml` | OpenAPI 3.0 |
| 数据模型 | `docs/models/` | Markdown/JSON |
| 实现时间线 | 嵌入TDS | 时间线 |

## 质量检查清单

- [ ] 所有FS功能已映射到TDS章节
- [ ] 每个功能有实现步骤
- [ ] 每个步骤有验收标准
- [ ] 技术栈已明确定义
- [ ] 数据模型已指定
- [ ] 已处理非功能需求
- [ ] 需求分析师已批准

## 常见陷阱

| 陷阱 | 预防措施 |
|------|----------|
| FS功能在TDS中遗漏 | 使用检查清单验证每个功能 |
| 实现步骤模糊 | 包含具体的验收标准 |
| 技术栈不匹配 | 首先与技术团队确认技术栈 |
| 过度设计 | 保持范围与FS对齐 |
